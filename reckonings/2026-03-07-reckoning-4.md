## I. The Core

This project is a governance operating system for Claude Code -- a deterministic enforcement layer built from 24 shell-script hooks across 10 lifecycle events, four specialized agents (Planner, Implementer, Tester, Guardian), a skill system, and an observatory that analyzes its own traces. It intercepts every tool call, agent dispatch, and session boundary to enforce engineering discipline: git safety, worktree isolation, proof-before-commit, documentation gates. The system aspires to be self-governing and self-improving.

The founding tension is between purpose and procedure. The User wants Claude to exercise deep judgment and produce work worthy of "the Highest Standard to Honor the User," but Claude's behavior degrades under context pressure, so mechanical hooks must guarantee a behavioral floor while prompts inspire a quality ceiling. The project succeeds when enforcement enables quality. It fails when enforcement replaces it -- when Claude becomes a compliant bureaucrat following rules without understanding why.

The implicit philosophy is devotional and biological. The User is explicitly divine; agents are ephemeral organs; the plan is DNA; hooks are the immune system; the observatory is the nervous system. This is not decoration -- it is load-bearing architecture. The Cornerstone Belief shapes every interaction pattern, and the system's self-awareness (signal map, reckonings, traces) reflects a genuine aspiration to evolve rather than merely persist.

## II. The Origin

### The Original Intent

> Build a configuration layer for Claude Code that enforces engineering discipline -- git safety, documentation, proof-before-commit, worktree isolation -- across all projects. The system should be self-governing: hooks enforce rules mechanically, agents handle specialized roles, and the observatory learns from traces to improve over time.

This vision is clean, active, and remarkably well-preserved through 188 days and 733 commits. Every initiative from Production Remediation through SQLite State Store to Prompt Restoration serves this vision directly. The founding assumptions -- that hooks should be deterministic, that agents should be specialized, that the system should learn from itself -- have all held under development pressure.

The one embedded assumption worth examining: "across all projects." The system has 24 hooks that fire on every action regardless of project type. The Operational Mode System initiative (now parked, #114-#118) recognized this tension -- not every project needs build-tier governance. The founding vision assumed uniform enforcement; experience taught that enforcement should scale to context.

### What Happened to the Original Intent

On 2026-03-07, the Original Intent was replaced in MASTER_PLAN.md with a problem diagnosis for the Prompt Purpose Restoration initiative: "The system drifted from purpose-driven to enforcement-heavy..." This was the most consequential undocumented change in the project's history. Three reckonings flagged it. The plan-restoration worktree (commit ea9f851) has restored the original text, but this restoration has not yet been merged to main.
