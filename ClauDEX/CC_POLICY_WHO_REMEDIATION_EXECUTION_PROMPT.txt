CHECKPOINT REPORT STAGED-SCOPE AUTHORITY RULE:
- Every checkpoint-stewardship / guardian-land report MUST derive its
  `Included scope` from the live output of `git diff --cached --name-only`
  at the time the report is assembled. Scope MUST NOT be narrated from
  memory, from a dispatch-context list, from a previous turn's scope list,
  or from a commit-message draft.
- Every such report MUST cite the exact current value of
  `git diff --cached --name-only | wc -l` as the count of included paths.
  If that count does not match the report's enumerated scope length, the
  report is invalid.
- Any path appearing in the `Included scope` cell of a report that is NOT
  present in `git diff --cached --name-only` is invalid and must be
  removed before the report is emitted. Guardian subagents that cannot
  obtain live `git` output for scope verification must STOP and report
  the inability, not substitute a recalled list.
- This rule was added 2026-04-17 in response to a scope-narration drift
  observed in instruction `1776422371697-0001-b39h0v` where a guardian
  subagent report's `Included scope` cell listed files that were not in
  the staged index. Mechanical pin:
  `tests/runtime/test_handoff_artifact_path_invariants.py::TestCheckpointReportScopeAuthority`.

CHECKPOINT REPORT EXCLUDED-SCOPE AUTHORITY RULE:
- Every checkpoint-stewardship / guardian-land report MUST derive its
  `Excluded scope` from the live output of `git status --short` at the
  time the report is assembled. Excluded scope MUST NOT be narrated
  from a session-start status snapshot, from memory, from a prior
  turn's status output, or from a recalled list of worktree mods.
- The report MUST explicitly distinguish staged entries (lines whose
  first column is non-space: `M `, `A `, `D `, `R `, `C `) from
  unstaged / untracked entries (lines whose first column is a space
  or `?`: ` M`, ` D`, `??`). Only unstaged / untracked entries belong
  in `Excluded scope`. Staged entries belong in `Included scope` and
  MUST be derived from `git diff --cached --name-only` per the
  staged-scope rule above.
- If `git status --short` shows no unstaged / untracked entries other
  than known lane-local artifacts (e.g., `?? .claudex/`), the report
  MUST say so explicitly with a single line such as
  `Excluded scope: none outside lane-local artifacts (`.claudex/`).`
  It MUST NOT enumerate unrelated or stale mods to create the
  appearance of a larger excluded surface.
- Any path appearing in the `Excluded scope` cell of a report that is
  NOT present in `git status --short` at report time is invalid and
  must be removed before the report is emitted. Guardian or
  orchestrator authors that cannot obtain live `git status --short`
  output for excluded-scope verification must STOP and report the
  inability, not substitute a recalled list.
- This rule was added 2026-04-17 in response to an excluded-scope
  narration drift observed in instruction `1776423808293-0006-4eqliy`
  where the checkpoint-retry report's `Excluded scope` cell listed
  many modified / untracked paths narrated from a session-start
  `gitStatus` snapshot rather than live `git status --short`
  (live state at report time showed only `?? .claudex/` outside the
  staged bundle). Mechanical pin:
  `tests/runtime/test_handoff_artifact_path_invariants.py::TestCheckpointReportExcludedScopeAuthority`.
